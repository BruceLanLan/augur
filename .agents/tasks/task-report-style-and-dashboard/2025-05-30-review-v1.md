# Report neutralization and CLI consensus output

The report module (`report.py`) was rebranded from a Buffett-centric "executive summary" to a neutral "18位大师共识裁决" framing. Two new analytical sections were added — a disagreement analysis that surfaces bull/bear factions and a structured 多空辩论 debate section. The CLI `analyze` command was rewritten from a simple per-agent table to a richer consensus summary box that matches the README's advertised output format.

Watch for:
- The CLI now runs `DecisionCoordinator.analyze_with_all()` (ThreadPoolExecutor, 30s timeouts) on every `augur analyze` invocation, and errors from `get_consensus()` are uncaught — an all-error scenario will traceback to the user (confirmed).
- `_format_disagreement_section` contains hardcoded narratives ("价值派 vs 风控派") that fire based solely on THEME_GROUPS membership, regardless of whether that's the actual root of disagreement (confirmed).
- AGENT_PHILOSOPHY dict must stay manually synchronized with the registry's agent IDs — no validation exists (confirmed).

## High-level view

The report sections (`_format_disagreement_section`, `_format_bull_bear_debate`) derive structure from the existing `THEME_GROUPS` mapping and score-based sorting rather than actual semantic disagreement analysis. The disagreement narratives will almost always fire the same two canned paragraphs for any stock with a typical agent signal distribution.

The CLI overhaul replaces a lightweight per-agent table with the full consensus pipeline (coordinator → parallel analysis → weighted consensus → formatted output). This is functionally the same code path as `augur consensus`, meaning `augur analyze` without `--persona` now duplicates that command's workload without clear differentiation.

Test coverage adds 9 new methods but uses a fixture with only 1 BEARISH agent out of 18, limiting confidence that multi-bear and balanced-split scenarios work correctly. CLI test assertions are shallow (4 string checks).

<summary>Issues (5)</summary>

1. **No error handling around DecisionCoordinator in CLI** — `coordinator.analyze_with_all()` and `get_consensus()` are called without try/except in the `analyze` command's non-JSON branch. A failure in consensus computation (e.g., all agents error out) will produce an unhandled exception traceback. Wrap in try/except and emit a user-friendly error.
2. **Hardcoded disagreement narratives fire on group membership, not actual reasoning** — The "价值派 vs 风控派" and "成长派 vs 保守派" labels appear whenever a bull happens to be in the 价值投资 group and a bear in 风险与宏观, regardless of whether those frameworks actually explain the disagreement. With 18 agents across 4 groups, both conditions fire for almost every stock. Consider gating on reasoning content or requiring a minimum faction size.
3. **AGENT_PHILOSOPHY must be manually synced with registry** — If an agent is added/removed/renamed in the registry but not updated in this dict, the philosophy annotation silently disappears. Custom personas created via the dashboard's hot-reload flow will never have entries. Add a startup assertion or derive from agent metadata.
4. **Test fixture has only 1 BEARISH agent** — The disagreement section's logic branches are tested with a lopsided distribution (12 bullish, 5 neutral, 1 bearish). Edge cases like even split, all-bearish, or 2-agent-only runs are untested.
5. **CLI `analyze` now duplicates `consensus` command** — Without `--persona`, `augur analyze` runs the identical coordinator pipeline as `augur consensus`. Consider documenting the distinction or unifying the commands.

## CLI analyze: uncaught consensus failure

The old `analyze` command had per-agent `try/except` blocks that emitted `"ERROR"` table rows. The new code delegates agent-level error handling to `DecisionCoordinator` (which wraps each agent in try/except), but `get_consensus()` itself can raise if all results have `SignalType.ERROR` — the weighted average computation would divide by zero. No guard exists at the CLI level:

```python
coordinator = DecisionCoordinator(registry)
results = coordinator.analyze_with_all(ctx)
consensus = coordinator.get_consensus(results, ticker=ticker.upper(), context=ctx)
```

Minor: `suffix = f"..."` (used 3 times) is a needless f-string.

## Disagreement section: always-true conditions

The narrative conditions check whether *any* bullish agent exists in 价值投资 and *any* bearish/neutral in 风险与宏观. With 7 agents in 价值投资 (most tending bullish) and 4 in 风险与宏观 (often cautious), both narrative blocks will fire for virtually any stock:

```python
value_bulls = [v for k, v in bullish_agents.items() if k in THEME_GROUPS.get("价值投资", {}).get("agents", [])]
risk_bears = [v for k, v in bearish_agents.items() if k in THEME_GROUPS.get("风险与宏观", {}).get("agents", [])]
if value_bulls and (risk_bears or risk_neutrals):
    lines.append("- **价值派 vs 风控派**: ...")
```

The fallback ("各Agent基于不同投资框架得出了不同结论") fires only when zero agents land in the inspected groups — effectively dead code given the 18-agent roster.

## Bull/bear debate: NEUTRAL agents presented as bears

`_format_bull_bear_debate` sorts all valid agents by score and takes the bottom 3 as "bears" regardless of their actual signal. When the bottom 3 are NEUTRAL (e.g., soros at 5.5, serenity at 5.8, dalio at 6.0), they're still rendered under "🔴 空方论点 (最谨慎的3位大师)" with bear emoji. The label says "most cautious" which is defensible, but the red-circle framing misrepresents their actual NEUTRAL position.

## Test gaps

Not tested in the CLI: Kelly sizing output, key findings rendering, signal distribution lines, or behavior when `consensus.key_findings` is empty/None. A regression in any of these would pass the current test suite.
