# Dashboard, bot, and coordinator hardening pass

This change hardens three areas of the Augur system: the dashboard frontend (XSS fix, polling lifecycle, accessibility), bot integrations (shared ticker extraction with stop-word filtering, thread-safe token refresh, exception guards), and the DecisionCoordinator (parallel analysis via ThreadPoolExecutor). A shared `bots/utils.py` module extracts common logic previously duplicated across four bots. Config module gains logging for malformed files and nested-path overwrites. 19 new tests cover the additions.

**Watch for:** The parallel coordinator catches per-agent exceptions but embeds the raw exception message in the `reasoning` field returned to the frontend, which could leak internal details through the API (likely). The stop-word list is a coarse filter that will false-negative on legitimate tickers that happen to be English words (e.g., "ALL" for Allstate) — a known trade-off acknowledged in the feature spec but worth noting for user-facing behavior (confirmed). The bot refactor imports `STOP_WORDS` from the shared module but each bot still reimplements the extraction loop locally rather than calling `utils.extract_ticker` — the deduplication is incomplete (confirmed).

## High-level view

The XSS fix in `signals.html` replaces inline `onclick="removeTicker('...')"` string interpolation with event delegation on `data-ticker` attributes, using the `escapeHtml()` function defined in `base.html`. This eliminates the injection surface where a malicious ticker stored in the watchlist could execute arbitrary JavaScript. The same template adds Page Visibility API hooks to pause and resume the auto-refresh interval, preventing unnecessary network traffic from background tabs.

The bot hardening introduces `src/augur/bots/utils.py` as a single source of truth for ticker extraction logic. However, each bot still defines its own `_extract_ticker` that reimplements the same findall-iterate-filter loop using the imported `STOP_WORDS` constant — the shared `utils.extract_ticker` function exists but isn't called by any bot. The Lark bot's token refresh is now protected by a `threading.Lock`, and the Telegram bot's `handle_message` now wraps the analyze call in try/except.

The coordinator change swaps the sequential agent loop for `ThreadPoolExecutor` with `max_workers=min(len(agents), 8)`. A blanket `except Exception` fallback reverts to sequential if threading itself fails. The per-agent `_analyze_single` wrapper catches exceptions and returns an `ERROR`-signal `AgentResponse` instead of raising.

Config validation is minimal: two `logger.warning` calls — one for YAML parse failures, one for non-dict parse results — and a warning when `set_config` overwrites a non-dict intermediate with a dict.

## XSS fix and event delegation in signals.html

The original `removeTicker` call used string interpolation directly into an `onclick` attribute: `onclick="removeTicker('${ticker}')"`. A ticker value like `'); alert('xss` would break out of the string and execute arbitrary code. The fix moves to a `data-ticker` attribute on the button element with `escapeHtml(ticker)` encoding (defined in `base.html`), and a single delegated click listener on `#watchlist-body` that reads the attribute value.

## Visibility API inconsistency between templates

Both `base.html` (health check) and `signals.html` (auto-refresh) pause their `setInterval` timers when `document.hidden` becomes true. The health check fires an immediate `checkHealth()` on resume before restarting the interval, giving instant UI feedback after tab switch. The signals page simply calls `startAutoRefresh()` on visibility restore without an immediate data fetch — tabs returning from background won't see fresh data until the next interval tick. Minor UX inconsistency.

## Shared stop-word filtering across bots

The stop-word set (`STOP_WORDS`) is a frozen set of ~30 common English words. The regex minimum was raised from 1 to 2 characters, eliminating single-letter false positives. Each bot still defines its own `_extract_ticker` function that reimplements the same loop: `re.findall` → iterate → check `STOP_WORDS` → return first match. The shared `utils.py` exports `extract_ticker` that does the identical thing, but only test code calls it. This means four copies of the logic exist in production, differing only in import path and docstring — a future maintenance hazard.

The ticker "ALL" (Allstate Corp, S&P 500 component) is in the stop-word list and will be filtered in every bot and the shared utility.

## Thread-safe token refresh in Lark bot

The `_get_tenant_access_token` method wraps the check-and-refresh logic in `with self._token_lock:`. The lock is a plain `threading.Lock()` (not RLock). The token refresh does network I/O (`urllib.request.urlopen` with 10s timeout) inside the lock, meaning concurrent requests serialize through a potentially slow path at token expiry. For a token cached ~2 hours this is rare, but under burst traffic at the refresh boundary all handler threads queue behind the lock for the duration of the Feishu API round-trip.

## Exception guard in Telegram handle_message and reasoning field exposure

The Telegram `handle_message` fix wraps the entire analysis path in try/except and replies with `f"❌ 分析失败: {e}"`. This prevents unhandled exceptions from propagating to the framework. The same pattern appears in the coordinator's `_analyze_single` wrapper: `reasoning=f"Analysis failed: {e}"`.

In both cases, `str(e)` is the raw exception message. For exceptions from deep library calls (e.g., file-not-found with full path, network errors with internal URLs), this exposes internals. The Telegram case sends it to users; the coordinator case sends it to the API consumer. Consider truncating to a generic message or first line only.

## Parallel analysis in DecisionCoordinator

`analyze_with_all` submits each agent to `ThreadPoolExecutor(max_workers=min(len(agents), 8))`. Results are collected via `as_completed`. If a future raises, it's caught and converted to an ERROR AgentResponse. A blanket `except Exception` around the executor block falls back to sequential.

No `timeout` is passed to `future.result()`. If an agent's `analyze` method hangs (network wait, infinite loop in custom YAML persona logic), the entire `as_completed` iteration blocks indefinitely. The sequential fallback has the same issue, so this isn't a regression, but parallel execution makes it more impactful: one hung agent now blocks the response for all 18 agents simultaneously rather than just delaying subsequent ones.

## Test coverage gaps

Not tested: the Page Visibility API integration (requires browser), the event delegation click handler in signals.html, the Lark token lock under concurrent access, the sequential fallback path when ThreadPoolExecutor itself fails, the WeChat login timeout parameter, and the Telegram `handle_message` exception guard path.
