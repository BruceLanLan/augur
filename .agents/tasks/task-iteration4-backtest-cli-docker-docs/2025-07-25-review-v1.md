# Backtest determinism, CLI plumbing, Docker hardening, doc corrections

This change sweeps across several subsystems with small, targeted fixes: the backtest sample-data generator switches from Python's non-deterministic `hash()` to a stable `sum(ord())` seed, an early-return guard prevents crashes on empty data, the `analyze` and `watchlist-add` CLI commands now properly thread sector/industry through to their models, Docker gets a non-root user and healthchecks, and documentation corrects an inflated command count. Eighteen new tests cover the additions.

**Watch for:** The Dockerfile runs as user `augur` but docker-compose.yml still mounts volumes to `/root/.augur` — containers will fail to write config at runtime (**confirmed**). The Dockerfile's CMD serves on port 8000 but labels itself "api", creating ambiguity with the compose `api` service on 8900 and the HEALTHCHECK that also targets 8000 (**confirmed**).

## High-level view

The backtest seed fix swaps `hash(ticker)` for `sum(ord(c) for c in ticker)`, eliminating cross-run variance caused by Python's `PYTHONHASHSEED` randomization. The empty-data guard returns a zero-value `BacktestResult` before importing the heavy registry machinery.

The CLI fix pipes `sector` and `industry` into the `MarketContext` constructor in `analyze_cmd`, closing a silent-drop bug where the user could pass `--sector Technology` and the value would vanish. The `watchlist-add` command gets the same options, storing them into the metrics dict alongside the numeric fields.

The Docker non-root user (`augur`) is created properly, but the accompanying volume mounts in docker-compose.yml were not updated to match — every service still mounts `augur-config` to `/root/.augur`, a path the `augur` user cannot access. This is the only blocking concern in the change.

Healthchecks on dashboard and api services use `urllib.request` in a Python one-liner, which is heavier than `curl` but avoids adding a system dependency to the slim image. Command count corrected from 20 to 17, `httpx` synced into requirements.txt, Makefile gets comment annotations.

## Volume mount vs non-root user conflict

The Dockerfile creates user `augur` with home directory `/app` and runs `mkdir -p /app/.augur` before switching to that user. At build time this is correct — `/app/.augur` is owned by `augur:augur`. But docker-compose.yml mounts the named volume `augur-config` at `/root/.augur` in every service:

```yaml
volumes:
  - augur-config:/root/.augur
```

The `augur` user has no access to `/root/`. Any code path that reads or writes `~/.augur` (config persistence, watchlist files, JSONL records) will fail with a permission error at runtime. The fix is to change the mount target to `/app/.augur` across all services, matching the path the Dockerfile creates.

## Dockerfile CMD and service identity

The Dockerfile ends with:

```dockerfile
# Default: start dashboard
CMD ["augur", "api", "--port", "8000", "--host", "0.0.0.0"]
```

The comment says "dashboard" but the command starts the API server. The compose `dashboard` service overrides this with `python -m dashboard.app --port 8000 --cors`, and the compose `api` service uses `augur api --port 8900`. If someone runs `docker run` without compose, they get the api app on 8000 — which coincides with the port the compose dashboard occupies. The HEALTHCHECK hits `/health` on 8000, which the api app does expose, so standalone usage works. The ambiguity is cosmetic but will confuse operators.

## Global RNG state in generate_sample_data

```python
random.seed(sum(ord(c) for c in ticker) % 10000)
```

This sets the module-level `random` seed. The determinism guarantee only holds if nothing else touches `random` in between calls. For demo/test usage this is acceptable, but for concurrent or interleaved usage (e.g., a server endpoint that generates sample data for multiple tickers in the same request), results become ordering-dependent. Using a local `Random` instance (`rng = random.Random(seed)`) would isolate the state without changing the API.

## Sector/industry default asymmetry

The `analyze` command declares `--sector` with `default=""` while `watchlist-add` uses `default=None`. The `analyze_cmd` then applies `sector or ""` — redundant given the Click default is already `""`. In `watchlist-add`, passing `--sector ""` stores an empty string in the metrics dict (the `if sector is not None` check passes for empty strings). This asymmetry doesn't cause a bug today but could surprise future maintainers expecting consistent semantics across commands.

## Test coverage gaps

The Docker tests read files from absolute paths (`/projects/sandbox/augur/Dockerfile`), which means they only pass in this specific workspace layout — they'd fail in CI or another developer's checkout.

Not tested: the runtime behavior of the non-root Docker user (no Docker daemon available), the actual healthcheck HTTP response, or the `watchlist-add` persistence path with sector/industry values surviving a round-trip through YAML.
