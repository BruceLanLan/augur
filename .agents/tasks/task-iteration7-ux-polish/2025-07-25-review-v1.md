# CLI output beautification, error consistency, and graceful degradation

This change adds three user-facing polish features to the Augur CLI and API: a `--no-color` flag backed by a `cli_format.py` module that provides formatted tables and bordered boxes; a global exception handler on both FastAPI apps to prevent stack trace leakage; and an `optional_deps.py` module that centralizes missing-dependency messaging with actionable `pip install` instructions. The approach wires these utilities into existing commands without changing core analysis logic.

**Watch for:** The `errors.py` module is dead code — defined and tested but never imported by any production path (confirmed). The `is_available()` check in the dashboard performs a real `importlib.import_module("augur.data")` on every request, which has a side-effect cost when the module *is* installed (likely). The NO_COLOR flag mutates `os.environ` globally, which can bleed state between commands in the same process (confirmed in test runner context).

## High-level view

The CLI formatting layer (`cli_format.py`) provides four helpers — `color_text`, `signal_icon`, `format_table`, `format_box` — that key off the `NO_COLOR` environment variable. The no-color detection is environment-based rather than context-based, which makes it simple but means it relies on process-global state that can leak across invocations.

The exception handling change adds identical `@app.exception_handler(Exception)` blocks to both `dashboard/app.py` and `src/augur/api.py`. These log the real error, then return a sanitized JSON envelope with `status`, `detail`, `timestamp`, and `path`. Existing endpoints also gain a `"status": "ok"` field for response consistency.

The optional-deps module (`optional_deps.py`) is a registry mapping package names to `(feature_description, install_command)` tuples, with `require_optional()` raising a descriptive `ImportError` and `is_available()` probing importability. Bot commands (telegram, slack, wechat, lark, cron-start) and the dashboard data endpoints now use this module instead of bare try/except ImportError blocks.

The `errors.py` module defines a standard error-response schema and a catalog of common error messages, but nothing in the production codebase imports it — it's dead code that will drift from actual usage.

## Dead `errors.py` module

`src/augur/errors.py` defines `error_response()`, `api_error_response()`, and a `ERRORS` catalog with entries like `"missing_yfinance"`, `"rate_limited"`, `"persona_not_found"`. None of these are imported anywhere outside `tests/test_iteration7.py::TestErrorsModule`. Meanwhile, the global exception handlers in both FastAPI apps construct the same `{"status": "error", "detail": ..., "timestamp": ..., "path": ...}` dict inline. The CLI's persona-not-found path uses raw `click.echo` strings. The module will drift from actual usage patterns since it is never exercised in production.

## NO_COLOR as environment mutation

```python
@click.group()
@click.option("--no-color", is_flag=True, ...)
@click.pass_context
def main(ctx, no_color):
    if no_color or os.environ.get("NO_COLOR", "") != "":
        os.environ["NO_COLOR"] = "1"
        ctx.obj["no_color"] = True
```

Setting `os.environ["NO_COLOR"]` is a process-wide side effect. The `cli_format.py` helpers then read `os.environ.get("NO_COLOR")` to decide formatting. In test suites using `CliRunner`, an earlier test that invokes `main --no-color` contaminates subsequent tests unless explicitly cleaned up. The tests in `test_cli.py` happen to work because the CliRunner creates an isolated environment — but if any test calls `main` *without* the `env` parameter after a `--no-color` run in the same process, it inherits the mutated env. A context-only approach (threading the flag through `ctx.obj` and passing it to format helpers) would be more robust.

## Import-on-request pattern in dashboard

```python
@app.get("/api/fetch/{ticker}")
async def api_fetch_ticker(ticker: str):
    from augur.optional_deps import is_available, get_install_hint
    if not is_available("augur.data"):
        ...
    from augur.data import fetch_market_context
```

The `is_available` call provides a nicer error message than a bare ImportError, but it's redundant with the subsequent import statement — if `augur.data` is missing, the `from augur.data import ...` would raise anyway. The pre-check exists solely for the custom HTTP 501 response. A module-level flag (`_HAS_DATA = is_available("augur.data")`) evaluated once at import time would achieve the same behavior without per-request overhead and without the semantic duplication.

## format_box off-by-one on title width

```python
if title:
    max_width = max(max_width, len(title) + 2)
# ...
top = f"+-- {title} " + "-" * (max_width - len(title) - 3) + "+"
```

When `max_width` equals `len(title) + 2`, the dash fill becomes `"-" * ((len(title) + 2) - len(title) - 3)` = `"-" * -1` = `""`. The box renders but the top border is one character shorter than the bottom. Fix: use `len(title) + 4` as the minimum, or clamp the fill to `max(0, ...)`.

## Test coverage gaps

Not tested: `format_box` and `format_table` output structure (no assertions on alignment or border characters), `clean_output` / `strip_emoji` functions, `color_text` helper, or the interaction between `NO_COLOR` env and `format_box` title rendering. The `errors.py` module is tested in isolation but since it's unused in production, those tests validate dead code.
