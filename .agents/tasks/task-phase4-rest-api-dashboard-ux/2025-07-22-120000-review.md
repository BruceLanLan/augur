# Config REST API and Dashboard UI/UX overhaul

Phase 4 introduces a full configuration management API (7 new endpoints) and reworks four dashboard pages: a functional settings page with per-persona model dropdowns, a personas page with search/filter/expand cards, an index page with quick-analysis widget and system status, and a stocks page with expandable reasoning and copy-to-clipboard. Supporting CSS adds responsive breakpoints, toast notifications, and hover/expand animations. A new test file covers the Config API endpoints.

Watch for: The custom-persona endpoint writes arbitrary YAML to disk with no path sanitization or content validation (confirmed). The config API has no authentication — anyone who can reach the server can overwrite model assignments and inject custom personas (confirmed). The `saveAll()` function fires 17 sequential PUT requests without batching (confirmed, minor UX issue).

## High-level view

The Config API is the backbone of the settings page: `get_config` / `set_config` / `save_config` from `augur.config` are exposed over HTTP with thin Pydantic wrappers. The approach is clean — one source of truth in YAML, in-memory cache, dot-notation updates — but the HTTP layer adds no access control or input validation beyond Pydantic type checks.

The custom-persona endpoint (`POST /api/custom-persona`) stands out as the highest-risk addition. It accepts a user-supplied `agent_id` that becomes a filename and writes arbitrary `yaml_content` to disk. A malicious `agent_id` containing `../` could write outside the intended directory.

The frontend work is purely additive CSS and vanilla JS. The personas page introduces a client-side filter that queries data attributes, the settings page calls PUT per-persona on save, and the stocks page adds expandable reasoning with a toggle button. All templates extend `base.html` which already provides a shared `showToast` utility.

Test coverage targets the happy path of the Config API (GET/PUT config, persona config, models, schema, settings page render). Destructive operations (custom-persona), error paths, and invalid-input scenarios are not tested.

<details>
<summary>Issues (5)</summary>

1. **Path traversal in custom-persona** — `agent_id` is used directly as a filename (`f"{body.agent_id}.yaml"`). An input like `../../etc/cron.d/malicious` writes outside `personas/custom/`. Sanitize by rejecting any `agent_id` containing `/`, `\`, or `..`, or by stripping to `[a-z0-9_-]`.
2. **No YAML validation on custom-persona content** — `yaml_content` is written verbatim. Malformed YAML will crash downstream loaders. Parse with `yaml.safe_load` before writing and return 400 on failure.
3. **No authentication on config write endpoints** — PUT /api/config, PUT /api/config/persona/{id}, and POST /api/custom-persona are world-writable when the server is network-accessible. At minimum, gate behind a shared secret header or localhost-only binding.
4. **N+1 saves on saveAll()** — Each persona PUT calls `save_config()` which rewrites `~/.augur/config.yaml`. With 17 personas that's 17 disk writes in rapid succession. Consider a batch endpoint or debounce saves server-side.
5. **No test coverage for custom-persona or error paths** — The test file covers only happy-path GET/PUT. No tests for invalid `agent_id`, nonexistent persona lookups on PUT, or filesystem failure modes.

## Path traversal and arbitrary file write via custom-persona

The `POST /api/custom-persona` handler constructs a file path from user input without sanitization:

```python
filepath = custom_dir / f"{body.agent_id}.yaml"
filepath.write_text(body.yaml_content, encoding="utf-8")
```

`body.agent_id` comes directly from the request JSON. If it contains `../../somewhere/file`, `Path` resolution will navigate out of `personas/custom/`. The `yaml_content` field is written as-is, meaning the endpoint is a generic file-write primitive for anyone with network access.

Mitigation: validate `agent_id` against a strict pattern (alphanumeric, hyphens, underscores only), and parse `yaml_content` with `yaml.safe_load` before writing.

## Config API: no authentication, no agent_id existence check

All config endpoints are unauthenticated. The CORS middleware (when `--cors` is passed) sets `allow_origins=["*"]`, meaning any browser tab can call PUT endpoints. For a local-development tool this may be intentional, but the `--cors` flag is designed for "Hermes integration" (the startup message says so), suggesting remote access is a use case.

Additionally, `PUT /api/config/persona/{agent_id}` accepts any string as `agent_id` — it doesn't verify the persona exists in the registry. This means typos silently create orphan entries in `per_agent` config that never get used.

## saveAll() serialization and disk thrashing

The settings page `saveAll()` function fires 17 concurrent `fetch` calls, each hitting `PUT /api/config/persona/{id}`. Each handler calls `save_config()` which rewrites the entire YAML file. Under concurrent execution this creates a race: two handlers read-modify-write the global `_config` dict simultaneously. Python's GIL makes individual dict operations atomic, but the sequence `set_config → save_config` across interleaved coroutines could produce a file where only the last writer's state survives. In practice with FastAPI's single-threaded event loop this is less dangerous, but the 17 sequential YAML serializations are wasteful. A `PUT /api/config` batch call or a debounced save would be cleaner.

## Index page quick-analysis: error handling gap

The quick-analysis widget on the index page calls `/api/analyze/{ticker}` and displays the consensus inline. On failure, it uses `alert('分析失败: ' + err)` — a blocking browser dialog that breaks the smooth UX of the rest of the page. The stocks page has the same pattern. Contrast with the settings page, which uses toast notifications for errors. The error UX is inconsistent across pages.

## Test coverage scope

The new `tests/test_api.py` covers:
- GET /api/config, GET /api/models, GET /api/config/persona/buffett
- PUT /api/config/persona/buffett with valid input
- GET /api/schema/persona
- GET /settings page render

Not tested:
- POST /api/custom-persona (the highest-risk endpoint)
- PUT /api/config with full body
- Invalid agent_id (nonexistent persona)
- Malformed request bodies (missing `model` field, etc.)
- Concurrent saves race condition
- Settings page behavior when `available_models` is empty

The existing test structure (pytest + TestClient + autouse fixture for config reset) makes adding these straightforward.
