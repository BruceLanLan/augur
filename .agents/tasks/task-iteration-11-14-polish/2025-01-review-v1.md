# Coordinator calibration, pipeline resilience, and polish bundle

This change touches the scoring hot path (sector-aware weight boosts, diversity penalty clamp, IC normalization), hardens the data layer (thread-safe cache, ticker validation, None coercion from yfinance), adds config/persona structural validation, introduces per-ticker API rate limiting, improves dashboard accessibility CSS, enriches CLI help text, and ships 41 new tests covering all of the above.

Watch for: `"ai" in industry_lower` matches any industry containing the substring "ai" — including "retail", "entertainment", "railroad" — incorrectly boosting tech personas for non-tech companies (confirmed); the diversity penalty's effective weight depends on dict iteration order, creating non-deterministic consensus scores (confirmed); rate limit state is per-process in-memory, meaning it resets on restart and doesn't coordinate across workers (confirmed).

## High-level view

The coordinator weight logic gains sector-awareness: tech tickers boost growth-oriented personas (cathie_wood, aschenbrenner, thiel) by 1.3x, financial-sector tickers boost value personas by 1.2x, healthcare boosts fisher. Rolling IC weights are normalized to sum=1 before the 50/50 blend with base weights. The diversity penalty is clamped via `max(0.3, ...)` so highly-correlated agent pairs never collapse a weight below 0.3x. However, the substring-based sector/industry matching has false positives that would misroute non-tech companies through the tech boost path.

The data pipeline wraps all cache operations in a `threading.Lock`, adds `_normalize_ticker()` as a single validation chokepoint, and applies the `or 0` defensive pattern to every yfinance info field.

Config validation checks top-level structure (defaults, per_agent, available_models) and warns on type mismatches. Persona YAML validation extends to agent_id regex, scoring_weights range, philosophy type, and threshold ordering.

Rate limiting uses a sliding-window list per ticker behind a Lock, 30 req/min/ticker, returning HTTP 429. Purely in-process state — doesn't survive restarts or coordinate across workers.

The test suite covers 200-iteration property tests on Kelly, thread-safety stress test with 10 threads x 50 ops, rate-limit integration tests via FastAPI TestClient, fuzz-style ticker parsing with injection/traversal/unicode payloads.

<summary>Issues (5)</summary>

1. **"ai" substring matches retail, entertainment, railroad** — `"ai" in industry_lower` is true for any industry containing the two-character sequence "ai" (ret**ai**l, entert**ai**nment, r**ai**lroad). This incorrectly boosts cathie_wood/aschenbrenner/thiel for non-tech companies. Use word boundary matching or check for "artificial intelligence" explicitly.
2. **"tech" substring matches "biotechnology"** — `"tech" in sector_lower` triggers the tech persona boost for biotech companies, which should route through healthcare. Use `sector_lower == "technology"` or exclude "bio" prefix.
3. **Diversity penalty is ordering-dependent** — The penalty accumulates multiplicatively over previously-processed agents (dict iteration order). The first agent processed always gets penalty=1.0; the last agent may get penalty as low as 0.3^N. This creates non-deterministic consensus scores depending on dict ordering.
4. **Rate limit doesn't survive restarts or scale across workers** — In-memory dict resets on deploy. Multiple uvicorn workers each maintain independent state. Document the limitation or use a shared store for production.
5. **No eviction of stale ticker keys in rate limit dict** — Per-ticker list entries are cleaned of expired timestamps, but the ticker key itself persists forever after first use. Unbounded over server lifetime.

## Sector-aware weight boosting and substring false positives

The sector boost logic uses substring matching on both sector and industry strings:

```python
is_tech = ("technology" in sector_lower or "tech" in sector_lower
           or "ai" in industry_lower or "semiconductor" in industry_lower
           or "software" in industry_lower)
```

The `"ai" in industry_lower` check is the most problematic. The two-character string "ai" appears as a substring in many common industry names: "Retail" (ret**ai**l), "Entertainment" (entert**ai**nment), "Railroad" (r**ai**lroad), "Containers & Packaging" (cont**ai**ners). Any stock in these industries would get cathie_wood, aschenbrenner, and thiel boosted 1.3x — materially distorting the consensus for a Walmart or Disney analysis.

The `"tech" in sector_lower` clause similarly matches "Biotechnology" as a sector name, though Yahoo Finance typically classifies biotech under "Healthcare" sector with "Biotechnology" as the industry. The industry substring "ai" is the higher-risk false positive since it affects a wider range of companies.

## Diversity penalty ordering asymmetry

```python
for processed_agent in adjusted_weights.keys():
    corr = corr_matrix.get(agent_id, {}).get(processed_agent, 0)
    if corr > 0.7:
        penalty *= max(0.3, 1.0 - (corr - 0.7))
```

`adjusted_weights` is populated as agents are iterated. The first agent has no prior entries to compare against, so it always gets penalty=1.0. The 18th agent is compared against all 17 prior entries. If 5 of those have correlation > 0.7, the penalty compounds: e.g., five pairs at corr=0.9 give `(max(0.3, 0.8))^5 = 0.33`, effectively reducing that agent's contribution to one-third. The first agent in iteration order with identical correlations would contribute at full weight.

In CPython 3.7+ dict iteration order is insertion order, so the ordering is stable but arbitrary (determined by the registration sequence in `_register_default_agents`). This means buffett (registered first) always gets full weight, while serenity (registered last) always gets maximum penalty. Whether this is intentional or accidental is unclear from the code.

## Rate limiting memory model

Requests exceeding the limit get a 429 without appending to the list, so per-ticker memory is bounded at 30 floats (240 bytes). The unbounded dimension is the number of distinct ticker keys. A server processing 50,000 unique tickers accumulates 50,000 dict entries (each holding an empty list after timestamps expire). Approximately 4MB worst case — unlikely to matter in practice but worth a periodic sweep if the server runs for months.
