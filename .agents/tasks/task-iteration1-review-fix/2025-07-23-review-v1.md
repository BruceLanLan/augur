# Defensive hardening across the Augur analysis pipeline

A batch of 12 fixes tightening edge-case handling across data ingestion, persona scoring, API validation, CLI output, and operational robustness (log rotation, cron error handling, timing metrics). The approach is conservative: guards around division-by-zero, lower-bound clamping on scores, regex validation on tickers, and best-effort rotation on disk files. Eight regression tests accompany the fixes.

Watch for:
- **dashboard/app.py still uses deprecated `datetime.utcnow()`** (confirmed) — the fix was applied only to the standalone `api.py`, not the dashboard.
- **Backtest rotation reads entire file into memory** (confirmed) — a 10MB JSONL file means a full `readlines()` load, which is fine for 10MB but the design doesn't cap the ceiling between rotation checks.
- **Cron error handling swallows all exceptions silently** (confirmed) — `print()` to stdout may not reach anyone in a headless cron environment.

## High-level view

The data-layer fix prevents a ZeroDivisionError when yfinance reports negative shareholder equity (debt_to_equity < 0). The guard is correct and the formula produces sane results for all positive inputs. The zero and negative cases now uniformly yield `debt_ratio = 0`, which is a reasonable "unknown" sentinel given the system treats 0 as "no data."

Persona score clamping applies `min(max(score, 0), 10)` uniformly across Buffett's five factors and Graham's three factors. This is a safe lower-bound fix — no factor formula can drive its score below zero. However, only two persona files were touched out of 18 registered agents; the remaining 16 may still have unclamped paths (though the test with extreme inputs suggests they don't produce negatives in practice).

The ticker validation regex in `api.py` mirrors what the dashboard already enforces, closing an injection vector on the standalone API. The `datetime.utcnow()` deprecation fix landed in `api.py` but was missed in `dashboard/app.py` (line 298), which still calls `datetime.datetime.utcnow()`.

The backtest rotation logic is best-effort: check size, read all lines, keep the last 5000. It runs synchronously inside `_save_records`, meaning rotation cost is paid on every write cycle when the file is large. The `except Exception: pass` is intentional (documented as best-effort) but means silent data loss on I/O errors during truncation.

The cron per-ticker try/except is a straightforward isolation pattern — one ticker crashing doesn't take down the batch. The error message goes to `print()`, which is consistent with the rest of the cron module's output style.

## Debt-ratio guard and the zero sentinel

The original code used `if debt_to_equity:` which is truthy for any non-zero value, including negative numbers. With `debt_to_equity = -150`, the formula computed `de_ratio = -1.5`, and `de_ratio / (1.0 + de_ratio)` = `-1.5 / -0.5` = `3.0` — a nonsense debt ratio exceeding 1.0. Worse, if yfinance returned exactly `-100` (equity of zero), `de_ratio = -1.0` and the denominator would be `0.0`, producing a `ZeroDivisionError`.

The fix changes the guard to `if debt_to_equity > 0`, which is correct. The else branch sets `debt_ratio = 0`. This means "company has negative equity" is indistinguishable from "data not available" — both produce 0. For the scoring personas, 0 is neutral (no penalty, no reward), which is a defensible choice. It might understate risk for companies that genuinely have negative equity, but that's an acceptable trade-off vs. crashing.

## Persona clamping coverage gap

The `min(max(score, 0), 10)` pattern was applied to:
- `buffett.py`: moat, earnings_predictability, financial_strength, management_quality, valuation (all 5 factors)
- `graham.py`: valuation, balance_sheet, earnings_stability (3 factors)

The test `test_no_negative_scores_extreme_inputs` exercises all 17+ registered personas with extreme values and asserts `result.score >= 0`. This provides integration-level assurance even for personas not explicitly clamped. However, the test checks the aggregate `score` field, not individual factor scores. A persona could have a negative factor that's outweighed by positive factors and still pass. The Buffett/Graham fixes are the only ones that guarantee per-factor non-negativity.

## Unfixed deprecation in dashboard/app.py

```python
# dashboard/app.py line 298 (unchanged in this PR)
"timestamp": __import__("datetime").datetime.utcnow().isoformat() + "Z",
```

The standalone `api.py` was updated to `datetime.now(datetime.timezone.utc).strftime(...)`, but the dashboard endpoint still uses the deprecated call. Python 3.12 emits a `DeprecationWarning`; future versions will remove it.

## Backtest rotation: correctness under concurrent access

The rotation sequence is: stat → open(read) → readlines → open(write) → writelines → open(append) → write new records. There's a TOCTOU window between the size check and the truncation write, and between truncation and the subsequent append. If two processes run `_save_records` concurrently (e.g., parallel backtest runs), one could truncate while the other is appending, corrupting the file. The bare `except Exception: pass` means this corruption is silently ignored.

For a single-user CLI tool this is likely fine. For a deployed service with concurrent requests to the backtest endpoint, it's a latent data-integrity risk.

## Cron error isolation vs. observability

The try/except in `cron.py` prints the error and continues — good for batch resilience. But the error message format `f"  Error analyzing {ticker}: {e}"` goes to stdout, not stderr, and not to the logging system. The rest of the PR adds `logging.getLogger(__name__)` to `registry.py` for similar diagnostic output. Using `logger.error()` in cron would keep the error in the same log stream as the registry debug messages and ensure cron schedulers (systemd, launchd) capture it.

## Test quality: source inspection vs. behavioral coverage

The backtest rotation test (`test_save_records_rotation_exists`) uses `inspect.getsource()` to check that the string `"10 * 1024 * 1024"` appears in the source code:

```python
source = inspect.getsource(bt._save_records)
assert "10 * 1024 * 1024" in source or "10_000_000" in source or "10MB" in source
```

This is a structural assertion, not a behavioral one. If the rotation logic is refactored (extracted to a helper, threshold moved to a constant, implementation changed to use `truncate()`), this test breaks even if the behavior is preserved. Conversely, if someone accidentally deletes the rotation body but leaves the comment `# 10MB`, the test passes.

The other new tests (debt ratio conversion, API validation, no-negative-scores) are properly behavioral.
