# Iteration 2 hardening: score clamping, input validation, XSS

A defensive-hardening pass across the Augur multi-agent system. Sixteen persona files gain `max(0, min(10, score))` clamping, Munger gets division-by-zero guards, Dayu gets a logic-bug fix in its momentum elif chain, the DecisionCoordinator gets a total_weight==0 guard plus final-score clamping, the MCP server gains ticker-regex validation and payload-size limits, and the dashboard templates get `escapeHtml()` before innerHTML writes. Twelve new pytest cases exercise the fixes.

**Watch for:** The `escapeHtml` call inside `renderMarkdown()` entity-encodes content *before* Markdown regex processing, which means any reasoning text containing `<`, `>`, or `&` (e.g. `P/E < 15`) will render as visible entity text in the browser — a cosmetic regression for all persona reasoning output (confirmed). The MCP ticker regex silently rejects tickers containing `^` or longer than 15 characters, which excludes some valid international symbols (likely). MCP validation tests only check the regex pattern in isolation, never the actual tool functions (confirmed).

## High-level view

Score clamping is mechanical and uniform. The only interesting persona-level fix is Dayu, where the elif chain for negative `change_pct` was reordered so the most-negative threshold is checked first, fixing a shadowed-branch bug that previously mapped all values below -5 into a single -1 penalty.

The coordinator hardens at two points: a zero-weight guard preventing `ZeroDivisionError` when all agents error, and a final clamp after meta-model blending and risk-manager veto (both can shift score outside [0, 10]).

MCP server validation uses a regex gate `^[A-Za-z0-9.\-]{1,15}$` on three tool entry points plus a 10 KB cap on YAML persona creation. Reasonable for US/HK tickers but restrictive for index symbols with caret prefixes.

The XSS fix adds `escapeHtml()` via DOM-based text-node creation and applies it to `agent_name` (correct) and inside `renderMarkdown()` (problematic — escaping before Markdown regex processing breaks rendering of content containing HTML-special characters).

<summary>Issues (4)</summary>

1. **escapeHtml breaks Markdown rendering of angle brackets** — `renderMarkdown()` calls `escapeHtml(md)` before applying regex replacements. Any reasoning text containing `<`, `>`, or `&` (common in financial comparisons: `P/E < 15`, `ROE > 20%`) will render as `&lt;`, `&gt;`, `&amp;` literally in the browser. Fix by escaping only untrusted fields (agent_name, user-created YAML persona reasoning) rather than blanket-escaping all Markdown content, or move to a DOMPurify-style approach. (confirmed)

2. **Ticker regex rejects index symbols and long tickers** — The pattern `^[A-Za-z0-9.\-]{1,15}$` rejects caret-prefixed symbols (`^GSPC`, `^DJI`) and any ticker over 15 characters. For a system supporting international markets, consider allowing `^` or making the pattern configurable. (likely)

3. **No integration test for MCP ticker rejection** — The test validates the regex via `re.match` in isolation but never calls `augur_analyze()` with an invalid ticker. A regression where validation is moved below `_build_context` would go undetected. Add a test that imports and calls the tool function directly. (confirmed)

4. **`escapeHtml` assumes browser DOM** — The function creates a `div` element. If this code is ever evaluated outside a browser (SSR, Web Worker), it will throw. Low risk for a Jinja2-rendered page but worth a comment. (possible)

## Dayu momentum elif chain fix

The original code checked `change_pct < -5` before `< -10` before `< -20`. Any value below -5 (including -15, -25) matched the first branch and received only a -1 penalty. The fix reverses the order to `< -20`, `< -10`, `< -5`, matching the positive side's descending-magnitude pattern. The tests confirm `-25` now scores strictly lower than `-7`.

## escapeHtml and Markdown interaction

```javascript
function renderMarkdown(md) {
    if (!md) return '';
    md = escapeHtml(md);  // <-- escapes BEFORE markdown regex
    return md
        .replace(/^#{1,3}\s+(.+)$/gm, '...')
        .replace(/\*\*(.+?)\*\*/g, '<strong>$1</strong>')
        ...
}
```

`escapeHtml` converts `<` to `&lt;` and `&` to `&amp;`. Persona reasoning regularly contains expressions like `P/E < 15` or `debt & equity`. These become `P/E &lt; 15` and `debt &amp; equity` in the rendered output — visible entity text rather than the intended characters. The `**bold**` and `## heading` patterns still work (their characters aren't HTML-special), but the bullet regex was updated to match `&bull;` since the literal `•` gets entity-encoded.

The correct fix is to escape only when the source is untrusted. Backend-generated reasoning from built-in personas is safe; only `augur_create_persona` YAML content is attacker-controlled.

## Coordinator total_weight==0 edge case

```python
if total_weight > 0:
    total_score /= total_weight
    total_confidence /= total_weight
else:
    total_score = 0.0
    total_confidence = 0.0
```

When all agents return `coverage_confidence=0` (all errored), every `adjusted_w` is zero. Without this guard the coordinator crashes. The else branch produces score=0, which flows to NEUTRAL via `max(signal_counts, ...)` — when all counts are 0.0, Python's `max` picks the first key arbitrarily (depends on dict insertion order). In practice the signal is meaningless for an all-error run, but if determinism matters, explicitly setting `consensus_signal = SignalType.NEUTRAL` in the else branch would be more intentional.

## Test coverage gaps

Not tested: actual MCP tool functions end-to-end (only regex pattern validated), `escapeHtml` behavior with angle-bracket content, coordinator when meta-model returns a score outside [0, 10], and Kelly sizing at boundary conditions (conf approaching 0.95 cap with score=10 yields pct=47.5% before the `min(20.0, ...)` clamp — worth verifying that clamp works).
