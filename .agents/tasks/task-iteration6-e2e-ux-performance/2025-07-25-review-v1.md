# E2E pipeline tests, data validation, and timing instrumentation

This iteration adds three new test suites (46 tests total) covering the end-to-end analysis pipeline, data pipeline edge cases, and dashboard error handling. It also adds timing instrumentation to `DecisionCoordinator` (measuring `analyze_with_all` and `get_consensus` latency) and `aria-live="polite"` attributes to dashboard error containers for screen-reader accessibility.

Watch for: The timing instrumentation stores `analysis_ms` in an instance variable (`_last_analysis_ms`) that is read by a later `get_consensus` call, creating a race condition under concurrent requests to the singleton coordinator (confirmed). Several data pipeline "tests" validate arithmetic formulas in isolation rather than exercising the actual `data.py` conversion code (confirmed). The `aria-live` additions are correct but minimal — only two containers are annotated.

## High-level view

The timing instrumentation uses `time.perf_counter()` to measure wall-clock latency in both `analyze_with_all` and `get_consensus`, writing the results into `consensus.metadata["timing_ms"]`. Because the dashboard uses a singleton `_coordinator`, concurrent requests will clobber `_last_analysis_ms` — the `analysis_ms` value reported in one request's consensus could belong to a different request.

The E2E test suite (10 tests) exercises the full pipeline through three entry points: the `AgentRegistry` coordinator directly, the Click CLI runner, and the FastAPI `TestClient`. These tests validate real code paths including score clamping, Kelly position sizing, error propagation, and type consistency.

The data pipeline suite (17 tests) mixes genuine integration tests (all-zero inputs, extreme values, mixed error handling, determinism, timing metadata, performance baseline) with unit tests that validate mathematical formulas in isolation. The formula tests (debt ratio conversion, unit conversions, ownership percentages) duplicate logic from `data.py` rather than calling through it, meaning they'll pass even if `data.py` changes its conversion logic.

The dashboard error handling suite (19 tests) validates ticker sanitization, HTML page loading, watchlist validation, persona 404, and custom persona YAML parsing via actual FastAPI endpoint requests.

On heavily loaded CI runners or resource-constrained containers, this can flake. Consider raising the threshold or marking it `@pytest.mark.slow` so CI can skip it when appropriate. (likely)

The watchlist mutation endpoint (`/api/watchlist/add`) modifies state and could leave residual effects. Consider using a session-scoped fixture instead. (likely)

## Timing instrumentation via `_last_analysis_ms` instance state

The `analyze_with_all` duration is stored as `self._last_analysis_ms`, then read back in `get_consensus` to populate `result.metadata["timing_ms"]`:

```python
# In analyze_with_all:
self._last_analysis_ms = elapsed_ms

# In get_consensus (called later):
result.metadata["timing_ms"] = {
    "analysis_ms": self._last_analysis_ms,  # stale if another request ran between
    "consensus_ms": consensus_ms,
}
```

The dashboard uses a module-level singleton `_coordinator` via `get_coordinator()`. Two concurrent requests hitting `/api/analyze/{ticker}` will both call `coord.analyze_with_all(ctx)` followed by `coord.get_consensus(...)` on the same instance. If request B's `analyze_with_all` finishes between request A's `analyze_with_all` and `get_consensus`, then A's timing metadata reports B's analysis duration. This is a data race on the instance variable — not a crash, but silently incorrect telemetry.

The `consensus_ms` measurement does not share this problem because `t0_consensus` is a local variable scoped to the single `get_consensus` call.

## Formula tests that don't exercise `data.py`

Three test classes (`TestDebtRatioConversion`, `TestUnitConversions`, `TestOwnershipConversion`) compute expected values inline:

```python
def test_conversion_162(self):
    debt_to_equity = 162
    de_ratio = debt_to_equity / 100.0
    debt_ratio = de_ratio / (1.0 + de_ratio)
    assert abs(debt_ratio - 0.618) < 0.01
```

This validates that the formula `de/(1+de)` produces the expected output, but does not import or call any function from `augur/data.py`. If someone changes `data.py` to handle `debtToEquity` differently (e.g., clamping negative values, using a different formula), these tests would still pass. They serve as documentation of intent but not as regression guards on the actual code path.

## Incomplete `aria-live` coverage

No `aria-live` was added to the watchlist error container or the settings page error states. If those containers are also populated dynamically via JavaScript (same pattern as the hero and stocks error panels), they should receive the same `aria-live="polite"` annotation.
