# Deep report generation from 18-agent consensus

This change adds a Chinese-language Markdown report generator (`src/augur/report.py`) that transforms multi-agent analysis results into a structured investment report. The report is exposed through three surfaces: a new CLI command `augur report TICKER`, an API endpoint `GET /api/report/{ticker}`, and a "生成深度报告" button on the stocks dashboard page. The approach groups agents into thematic investment clusters (value, growth, macro/risk, technical) and renders each cluster's findings into a cohesive narrative with scoring tables, a risk matrix, and position sizing recommendations.

**Watch for:** The report endpoint re-runs the full 18-agent analysis on every call rather than reusing the analysis already performed for the page (confirmed). The `hasUserMetrics()` check in the frontend uses a different field set than the backend `has_user_metrics` check, which can cause silent metric drops (likely). The `zhang_lei` agent is classified under "技术与量化" in the report module but under the China-specialist group in the feature spec, creating a classification discrepancy (confirmed).

## High-level view

The report module is a pure formatting function: it takes already-computed `AgentResponse` objects and a `MarketContext`, then assembles Markdown sections string-by-string. There are no side effects, no I/O, no caching. This means the API endpoint must re-drive the entire agent pipeline to produce a report, even when the user just ran an analysis moments earlier.

The API endpoint is a near-exact copy of the existing `/api/analyze/{ticker}` handler, with identical parameter lists, identical rate-limiting, and identical MarketContext construction. The only difference is the final step: it calls `generate_report()` instead of serializing raw agent responses. This duplicated ~80 lines of boilerplate creates a maintenance surface where the two endpoints can silently drift apart.

The frontend integration wires the report generation as a secondary action after analysis completes. The "查看完整分析" link on the homepage now appends `&report=1` to auto-trigger report generation on the stocks page, and a `setTimeout(generateReport, 500)` fires the request once results render. This creates a race condition window if `renderResults` completes but the DOM isn't yet interactive.

The test suite covers unit-level edge cases, CLI invocations, and the API endpoint, but does not test the frontend auto-report path or the interaction between analysis and report generation when `auto_fetch` succeeds.

1. **Duplicate analysis on report generation** (confirmed) — The `/api/report/{ticker}` endpoint re-runs the full 18-agent pipeline rather than accepting pre-computed results or caching. For users who just ran analysis, this doubles latency and LLM costs. Consider accepting analysis results as POST body or adding a short-lived cache keyed on ticker+context hash.
2. **Frontend-backend metric field mismatch** (likely) — `hasUserMetrics()` checks `inp-debt`, `inp-revg`, `inp-epsg` but `generateReport()` maps these to `debt_ratio`, `revenue_growth`, `earnings_growth`. However, `hasUserMetrics()` doesn't check `revenue_growth` or `debt_ratio` directly, so a user who only enters revenue growth will not trigger the "has metrics" path in JS, but the backend will see `revenue_growth != 0` and use manual mode. Align the detection logic.
3. **Agent classification inconsistency** (confirmed) — The feature spec lists `zhang_lei` as a China specialist alongside `duan_yongping`, `li_lu`, `dan_bin`, but `THEME_GROUPS` in `report.py` places `zhang_lei` under "技术与量化". This affects which theme section the agent's analysis appears in.
4. **Race condition on auto-report** (likely) — `setTimeout(generateReport, 500)` after `renderResults` assumes the DOM will be ready. If the browser's event loop is congested, `generateReport` could fire before the ticker input is populated. Call `generateReport()` synchronously at the end of `renderResults` instead.
5. **MarketContext construction duplicated three times** (confirmed) — The same ~20-field `MarketContext(...)` construction appears in `/api/analyze`, `/api/report`, and `cli.py report_cmd`. Any schema change requires updating all three. Extract a shared factory function.
6. **No response size bound on report** (possible) — `generate_report()` concatenates all agent findings and risks without a length cap. For tickers with verbose agent responses, the report could grow unboundedly. Check whether downstream consumers (clipboard, file download) handle large payloads gracefully.

### Duplicated analysis pipeline in the report endpoint

The `/api/report/{ticker}` handler duplicates the entire request-handling pattern from `/api/analyze/{ticker}`: same parameter list, same validation, same rate-limit call, same `has_user_metrics` logic, same MarketContext construction, same `coord.analyze_with_all()` + `coord.get_consensus()` sequence. The only divergence is the response payload (raw JSON vs. generated Markdown).

A user who clicks "生成深度报告" after completing analysis will wait for all 18 agents to run again. On a system where agents make LLM calls, this could mean 10-30 seconds of redundant computation. The endpoint doesn't accept pre-computed results as input, and there's no server-side cache of recent analyses.

A practical fix: either make the report endpoint accept a POST body containing the agent responses (the frontend already has `lastAnalysisData`), or introduce a short-TTL cache keyed on `(ticker, context_hash)` so the second call within a few minutes reuses results.

### Frontend metric forwarding asymmetry

In `stocks.html`, `generateReport()` builds query parameters:

```javascript
params.set('gross_margins', num('inp-gm') / 100);
params.set('revenue_growth', num('inp-revg') / 100);
params.set('debt_ratio', num('inp-debt') / 100);
```

But `hasUserMetrics()` checks a different subset:

```javascript
function hasUserMetrics() {
    return num('inp-price') > 0 || num('inp-pe') > 0 || num('inp-pb') > 0 ||
           num('inp-gm') > 0 || num('inp-roe') > 0 || num('inp-mktcap') > 0;
}
```

If a user enters only `inp-revg = 25` (meaning 25% growth), `hasUserMetrics()` returns false, so `auto_fetch` is sent as `true`, and the backend ignores the manual `revenue_growth=0.25` parameter because yfinance data takes precedence. The user's input is silently discarded.

### Agent theme classification

`THEME_GROUPS` in `report.py`:

```python
"技术与量化": {
    "agents": ["arps", "dayu", "zhang_lei"],
}
```

The feature spec (FEAT-001.json step 4) says: "China specialists (zhang_lei, duan_yongping, li_lu, dan_bin)". Zhang Lei (高瓴资本创始人) is primarily a value/growth investor in the China market, not a quant/technical analyst. His classification here affects where his analysis appears in the report and whether readers find it in the expected thematic section.

### Auto-report timing fragility

```javascript
if (window._autoReport) {
    window._autoReport = false;
    setTimeout(generateReport, 500);
}
```

The 500ms delay is a heuristic. If the browser's event loop is congested (mobile, low-end hardware), DOM updates may not have flushed by the time `generateReport` reads `inp-ticker`. More robust: call `generateReport()` synchronously at the end of `renderResults` (after all DOM writes) rather than using a timer. The ticker value is already set by the time `renderResults` executes, so the timeout serves no purpose.

### Test coverage gaps

What's not tested:

- The frontend JS `generateReport()` / `copyReport()` / `downloadReport()` functions (no integration tests)
- The interaction between `auto_fetch=true` and the report endpoint when `fetch_market_context` succeeds (the test uses `auto_fetch=false`)
- Concurrent report generation requests hitting the rate limiter
- Report output size under adversarial input (very long reasoning strings)
