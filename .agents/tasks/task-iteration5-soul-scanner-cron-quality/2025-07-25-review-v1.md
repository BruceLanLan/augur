# Security hardening, cron deep merge, and scanner backward-compat

This change closes three security holes (eval injection, directory traversal, format parameter injection), fixes a shallow-merge bug in watchlist config loading, adds concurrent-execution protection to the cron scheduler, and fills in 6 missing backward-compat exports in the scanner shim. The approach is sound across the board — AST whitelisting for eval, path resolution for traversal, pidfile for concurrency — but the traversal check has a prefix-collision weakness, and the `signal` import is dead code.

**Watch for:** Path traversal check bypasses via directory-name prefix collision (confirmed). Unused `import signal` in cron.py (confirmed). AST sandbox still calls `eval()` after validation — semantically correct but leaves a live `eval` in the codebase (confirmed, low risk).

## High-level view

The AST-based eval sandbox in `persona_loader.py` walks the parsed expression tree and rejects any node type outside a whitelist before calling `eval`. The whitelist is conservative (no `Call`, `Attribute`, `Subscript`, `ListComp`, etc.) and the name-check restricts identifiers to `MarketContext` fields plus boolean constants. After validation, the code still calls the original `eval()` — safe given the preceding check, but a future refactor that skips the AST walk would silently re-open the hole.

The directory traversal fix in `soul.py` restructures `inject_soul()` to compute the output path for all three formats before a single `resolve()` + `startswith()` gate. However, the `startswith` comparison uses raw string matching without appending a path separator, which means a directory named `/tmp/out` would incorrectly allow writes to `/tmp/output-evil/file.md`.

The cron module gains a recursive `_deep_merge()` helper that replaces the old shallow `dict.update()`, preserving nested defaults (e.g., timezone when only cron is overridden). Concurrency protection uses a pidfile with `os.kill(pid, 0)` probing to detect stale locks, and `max_instances=1` on the APScheduler job prevents overlapping analysis runs. The pidfile is self-healing (stale PIDs are detected) but `SIGTERM` isn't handled, so container stops leave stale pidfiles until the next startup.

In `scanner/personas/__init__.py`, `DayuAgent` is re-imported despite already being present in the base branch — a harmless duplicate worth deduplicating.

## AST eval sandbox in persona_loader

The original `_eval_rule_condition` was a textbook `eval()` with `__builtins__` stripped — a defense trivially bypassed via `().__class__.__bases__[0].__subclasses__()`. The new implementation parses the condition string into an AST and walks every node, rejecting anything outside a tuple of allowed types. Only `ast.Name` nodes whose `id` appears in the `MarketContext`-derived namespace pass through. This blocks function invocation, attribute traversal, subscript access, and f-string injection in one pass.

After validation, the code still invokes `eval(condition, {"__builtins__": {}}, ns)`. This is semantically safe — the AST check has already proven the expression contains only constants, names, comparisons, and arithmetic — but it means a future refactor that accidentally skips the AST check would re-open the hole. Compiling the validated tree directly (`compile(tree, "<condition>", "eval")`) and evaluating the resulting code object would make the safety guarantee structural rather than procedural.

## Directory traversal protection in soul.py

The refactored `inject_soul()` computes the `output_file` path for all three formats in one block, then runs a single traversal check:

```python
resolved_output = output_file.resolve()
resolved_out_dir = out_dir.resolve()
if not str(resolved_output).startswith(str(resolved_out_dir)):
    raise ValueError(...)
```

This has a prefix-collision flaw. If `out_dir` resolves to `/data/profiles` and an attacker supplies `profile_path = "../profiles-evil/payload"`, the resolved output would be `/data/profiles-evil/payload-soul.md`. The string `/data/profiles-evil/payload-soul.md` starts with `/data/profiles`, so the check passes and the file is written outside the intended directory.

The fix is to compare against `str(resolved_out_dir) + os.sep` (or equivalently, check `resolved_output.is_relative_to(resolved_out_dir)` on Python 3.9+). This is a real bypass, not theoretical — the `raw` and `claude` formats construct the filename as `out_dir / f"{profile_path}-soul.md"`, so a profile_path like `../<out_dir_name>-evil/x` would land outside. Verified with:

```python
out_dir = Path('/tmp/out')
file = Path('/tmp/output-evil/file.md')
str(file.resolve()).startswith(str(out_dir.resolve()))  # True — incorrect
str(file.resolve()).startswith(str(out_dir.resolve()) + '/')  # False — correct
```

## Cron concurrency and pidfile gap

The pidfile logic probes the recorded PID with `os.kill(pid, 0)` and treats `OSError` as a stale lock. This is self-healing: if the previous process died without cleanup, the next startup detects it and proceeds. But `SIGTERM` — the signal sent by `docker stop`, systemd `ExecStop`, and most process managers — is not handled. The `import signal` statement at line 398 is present but no handler is ever registered. On `SIGTERM`, the process dies without unlinking the pidfile.

Registering a handler would be trivial:

```python
def _cleanup(signum, frame):
    pidfile.unlink(missing_ok=True)
    raise SystemExit(0)

signal.signal(signal.SIGTERM, _cleanup)
```

This would also make the currently dead `signal` import non-dead.
