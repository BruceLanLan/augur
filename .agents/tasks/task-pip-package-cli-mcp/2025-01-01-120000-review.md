# Pip-installable package restructuring with CLI and MCP server

The Augur multi-agent investment analysis system is restructured from a flat `scanner/` directory into a proper `src/augur/` package with `pyproject.toml`, a Click CLI (`augur`), and a FastMCP-based MCP server. All original `scanner.*` import paths remain functional via thin shims that re-export from the new `augur.*` package. All 41 tests pass and the package installs cleanly.

**Watch for:**
- YAML custom personas (`da-yu.yaml`, `thiel.yaml`) share `agent_id` with hardcoded agents and silently overwrite them (confirmed) — this means they never add net-new agents, only replace implementations.
- `augur.registry` and `augur.personas.base` still import from `scanner.*` via try/except for advanced features (confirmed) — this creates a latent circular dependency path (`augur -> scanner -> augur`) that works today only because those scanner modules don't exist and the imports always fail.
- Config module uses module-level mutable globals without thread safety (confirmed), while the registry uses an `RLock` — inconsistency that can surface in the API server under concurrent requests.

## High-level view

The package layout follows `src/` convention with setuptools discovery. Import rewiring is clean — every agent file, the registry, and the persona loader now use `augur.*` internal imports, and all backward-compat paths are verified by dedicated tests.

The two existing YAML files (`thiel.yaml` with `agent_id: thiel`, `da-yu.yaml` with `agent_id: dayu`) collide with hardcoded agents and overwrite them. The total agent count stays at 17 rather than 19. The YAML versions use a generic `YamlAgent` with factor-rule evaluation rather than the specialized Python implementations, so two agents silently change behavior when custom YAML files are present.

The consensus engine in `augur.registry.DecisionCoordinator.get_consensus` attempts to load 8+ optional scanner submodules (industry_matrix, regime_weights, risk_manager, kelly_sizer, meta_model, probability_calibrator, rolling_ic, ten_x_screener) none of which exist in the repository. These import from `scanner.*` rather than `augur.*`, which means when those modules are eventually created under `augur.*`, the references here won't find them without updates.

The CLI and MCP server create a fresh `AgentRegistry` on every invocation (instantiating 17 agents + reading YAML files), despite a `get_registry()` singleton already existing. For the API server under uvicorn, this means redundant per-request initialization.

<summary>Issues (5)</summary>

1. **YAML persona agent_id collision** — `thiel.yaml` and `da-yu.yaml` use the same `agent_id` as hardcoded agents, silently replacing them instead of adding new agents. Either rename the YAML agent_ids to be distinct, or add explicit "override mode" semantics with logging.
2. **Latent circular dependency via scanner imports** — `augur.registry` and `augur.personas.base` import from `scanner.*` for optional features. Once those modules are implemented under `augur.*`, these references will be stale. Migrate them to `augur.*` imports now (still guarded by try/except) so the dependency only flows one way: `scanner -> augur`.
3. **Config module thread safety** — `augur.config` uses bare module globals (`_config`, `_config_path`) without locking, while `augur.registry` is thread-safe with RLock. Under concurrent API requests via uvicorn, `set_config` can race with `get_config`. Add a module-level lock or convert to a class.
4. **AgentRegistry instantiation on every request** — The MCP tools and CLI create a new `AgentRegistry()` per call. The existing `get_registry()` singleton exists but isn't used. For the API server this means per-request overhead instantiating 17 agents and reading YAML files.
5. **`requires-python = ">=3.8"` vs actual compatibility** — `pyproject.toml` claims Python 3.8+ but uses `str | Path` type union syntax (PEP 604) in `persona_loader.py`. The `from __future__ import annotations` makes this safe at runtime but misleading — static analysis and `get_type_hints()` will fail on 3.8. Either use `Union[str, Path]` or bump to `>=3.9`.

## YAML persona collision and registry initialization

The registry calls `_register_yaml_personas()` after hardcoding all 17 agents. It iterates `personas/custom/*.yaml` and inserts each into `self._agents[agent.agent_id]`. Since `thiel.yaml` declares `agent_id: thiel` and `da-yu.yaml` declares `agent_id: dayu`, both overwrite the hardcoded `ThielAgent` and `DayuAgent` instances. The YAML versions use a generic `YamlAgent` with factor-rule evaluation rather than the specialized Python implementations, so the behavior of these two agents changes depending on whether custom YAML files are present.

This is documented in the FEAT-001 findings field but not defended in code. There's no logging, no warning, and no way for a user to know the Python implementation was silently replaced.

## Forward-referencing scanner.* imports inside augur

`augur/registry.py` lines 152-176, 225, and lines 282-343 contain:

```python
from scanner.industry_matrix import detect_industry, get_agent_weights
from scanner.regime_weights import detect_regime, apply_regime_weights
from scanner.macro_features import fetch_macro_features
from scanner.regime_router import RegimeRouter
from scanner.rolling_ic import load_rolling_ic_weights
from scanner.probability_calibrator import calibrate_confidence
from scanner.meta_model import MetaModel
from scanner.risk_manager import RiskManager
from scanner.kelly_sizer import compute_position_size
from scanner.ten_x_screener import attach_ten_x_to_consensus
```

`augur/personas/base.py` line 207 and `augur/personas/lynch.py` line 44:
```python
from scanner.agent_hyperparams import load_optimized_thresholds
```

The concern is architectural direction: the new canonical package (`augur`) should not import from the shim layer (`scanner`). When these features are built, they should live in `augur.*` and these imports should point there. Currently the dependency graph is bidirectional (`scanner -> augur` via shims, `augur -> scanner` via these try/excepts), which will cause actual circular import failures the moment any of those scanner modules are created and themselves import from augur.

## Config path resolution after non-editable install

`augur.config._find_config_file()` searches: (1) `~/.augur/config.yaml`, (2) `Path(__file__).parent.parent.parent / "config" / "agents.yaml"` (three parents up from `src/augur/config.py` = repo root), (3) `Path.cwd() / "config" / "agents.yaml"`. After a non-editable `pip install`, path (2) resolves to inside site-packages, not the repo root. Only path (3) would work, and only if the user runs commands from the repo directory. For an installed package, the fallback to `~/.augur/config.yaml` (which doesn't exist initially) means config loads as an empty dict — all model assignments in `config/agents.yaml` are silently ignored.

This is fine for development (`pip install -e .`) but will be surprising for distribution.
